# LinuxHUD 项目定位、架构设计与实现方案

## 一、项目定位

### 核心定义
**LinuxHUD** 是一个基于 Linux DRM/KMS 的硬件级平视显示器守护进程，通过直接操作显卡的 Overlay Plane 实现绝对优先级的信息叠加显示。

### 设计哲学
- **硬件优先**：绕过整个图形栈（X11/Wayland），直接与显卡驱动对话
- **零妥协**：即使桌面环境卡死，HUD 依然能瞬间刷新
- **极简主义**：最小依赖，最小开销，最大控制权

### 应用场景
1. **AI 状态监控**：实时显示 AI 推理状态、系统资源占用
2. **关键信息覆盖**：在任何应用程序之上显示关键系统信息
3. **调试与诊断**：系统级调试信息的无干扰显示
4. **安全监控**：在锁屏状态下显示安全警报

## 二、架构设计

### 1. 分层架构

```
┌─────────────────────────────────────────────┐
│           用户态应用层 (AI/监控逻辑)          │
├─────────────────────────────────────────────┤
│           渲染引擎层 (Cairo/Pango)           │
├─────────────────────────────────────────────┤
│           内存管理层 (Dumb Buffer)           │
├─────────────────────────────────────────────┤
│           DRM/KMS 控制层 (libdrm)            │
├─────────────────────────────────────────────┤
│           内核态驱动层 (GPU Driver)          │
├─────────────────────────────────────────────┤
│           硬件层 (Overlay Plane)            │
└─────────────────────────────────────────────┘
```

### 2. 核心组件

#### 2.1 资源管理器 (ResourceManager)
- **职责**：发现和管理 GPU 资源
- **关键资源**：
  - Connector：物理显示接口（HDMI/DP/eDP）
  - CRTC：显示控制器
  - Plane：硬件叠加平面（Primary/Overlay/Cursor）
  - Framebuffer：显存缓冲区

#### 2.2 内存管理器 (MemoryManager)
- **职责**：管理 Dumb Buffer 的分配、映射和释放
- **关键操作**：
  - `DRM_IOCTL_MODE_CREATE_DUMB`：创建哑缓冲区
  - `mmap`：映射到用户态地址空间
  - 双缓冲切换：避免画面撕裂

#### 2.3 渲染引擎 (RenderEngine)
- **职责**：将文本/图像渲染到缓冲区
- **技术选型**：
  - Cairo：2D 矢量图形渲染
  - Pango：文本布局和渲染
  - 像素格式：ARGB8888（32位带Alpha通道）

#### 2.4 显示控制器 (DisplayController)
- **职责**：管理 Plane 的显示属性
- **关键操作**：
  - `drmModeSetPlane`：设置 Plane 位置和缓冲区
  - `drmModeSetProperty`：调整透明度、Z轴顺序
  - Page Flip：同步刷新

### 3. 数据流

```
[AI/监控数据] → [文本生成] → [Cairo渲染] → [Dumb Buffer] → [DRM Commit] → [硬件显示]
```

### 4. 状态机

```
┌─────────┐    ┌──────────┐    ┌──────────┐    ┌─────────┐
│  INIT   │───▶│ RESOURCE │───▶│  READY   │───▶│ RUNNING │
└─────────┘    │ DISCOVERY│    │          │    │         │
               └──────────┘    └──────────┘    └─────────┘
                    │               │               │
                    ▼               ▼               ▼
               [扫描DRM资源]  [分配Buffer]   [渲染循环]
```

## 三、技术选型

### 1. 编程语言：C

**选择理由**：
- **零开销抽象**：直接调用内核接口，无运行时开销
- **内存控制**：精确控制内存分配和释放
- **系统集成**：与 Linux 内核 API 天然契合
- **最小依赖**：避免引入复杂的运行时环境

**备选方案对比**：
| 语言 | 优势 | 劣势 |
|------|------|------|
| C | 零开销、直接控制 | 手动内存管理、开发效率较低 |
| Rust | 内存安全、现代特性 | 学习曲线、编译时间较长 |
| Go | 开发效率高 | CGO 开销、GC 暂停 |

### 2. 核心依赖库

| 库 | 版本 | 用途 |
|----|------|------|
| libdrm | 2.4.125+ | DRM/KMS 接口 |
| cairo | 1.18+ | 2D 图形渲染 |
| pango | 1.50+ | 文本布局 |
| glib | 2.78+ | 基础数据结构 |

### 3. 开发工具

- **构建系统**：Meson（现代化、跨平台）
- **调试工具**：valgrind、gdb、drm_info
- **测试框架**：自定义测试脚本

## 四、实现方案

### Phase 1：最小可行产品 (MVP)

#### 目标
在 Overlay Plane 上显示静态文本

#### 核心功能
1. 打开 `/dev/dri/card0`
2. 发现 Overlay Plane 资源
3. 创建 Dumb Buffer
4. 渲染简单文本
5. 提交到 Overlay Plane

#### 代码结构
```
linuxhud/
├── meson.build
├── src/
│   ├── main.c
│   ├── drm_device.c
│   ├── drm_device.h
│   ├── buffer.c
│   ├── buffer.h
│   ├── renderer.c
│   └── renderer.h
├── include/
│   └── linuxhud.h
├── tests/
│   └── test_overlay.c
└── docs/
    └── README.md
```

### Phase 2：动态更新

#### 目标
实现实时文本更新和动画

#### 核心功能
1. 双缓冲机制
2. 定时更新循环
3. 事件驱动架构
4. 信号处理（SIGTERM/SIGINT）

### Phase 3：AI 集成

#### 目标
与 AI 系统集成，显示推理状态

#### 核心功能
1. 与 AI 服务通信（Unix Socket/DBus）
2. 状态机管理
3. 配置文件支持
4. 日志系统

## 五、关键技术细节

### 1. Overlay Plane 选择策略

```c
// 优先级顺序
1. DRM_PLANE_TYPE_OVERLAY (硬件叠加平面)
2. DRM_PLANE_TYPE_CURSOR (硬件光标平面)
3. DRM_PLANE_TYPE_PRIMARY (主平面，需谨慎使用)
```

### 2. 像素格式支持

```c
// 推荐格式
DRM_FORMAT_ARGB8888  // 32位，带Alpha通道
DRM_FORMAT_XRGB8888  // 32位，无Alpha（性能略好）
```

### 3. 透明度控制

```c
// 通过 Plane 属性控制
drmModeObjectSetProperty(fd, plane_id, 
                        DRM_MODE_OBJECT_PLANE,
                        "alpha", alpha_value);
```

### 4. 错误处理策略

```c
// 优雅降级
1. Overlay Plane 不可用 → 尝试 Cursor Plane
2. Cursor Plane 不可用 → 使用 Primary Plane（覆盖模式）
3. 所有 Plane 不可用 → 回退到终端输出
```

## 六、安全与权限

### 1. 权限要求
- 用户必须属于 `video` 组
- 或使用 `root` 权限运行
- 或通过 `udev` 规则授权

### 2. 安全考虑
- 输入验证：防止缓冲区溢出
- 资源限制：限制 Plane 使用数量
- 信号处理：优雅关闭资源

## 七、性能指标

### 目标性能
- **刷新延迟**：< 16ms (60fps)
- **CPU 占用**：< 1%
- **内存占用**：< 10MB
- **启动时间**：< 100ms

### 性能优化
1. **避免不必要的重绘**：脏矩形检测
2. **预计算文本布局**：缓存 Pango 布局
3. **批量提交**：减少 DRM ioctl 调用
4. **内存对齐**：确保 Buffer 对齐到页面大小

## 九、风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| Overlay Plane 不可用 | 功能受限 | 支持 Cursor/Primary 回退 |
| 驱动兼容性 | 显示异常 | 抽象硬件差异层 |
| 权限问题 | 无法启动 | 提供权限配置脚本 |
| 资源泄漏 | 系统不稳定 | 严格的资源管理 |
